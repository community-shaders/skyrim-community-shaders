SAMPLE CONTRIBUTION - BACKGROUND RECOVERY
[McGuire2012] [McGuire2013] approximates scatter-as-you-gather as following:
Classify samples as background or foreground with respect to current pixel
Calculate a weight that depends on three conditions
Foreground sample with a velocity bigger or equal than its offset
Background sample within the bounds of current pixel velocity (allows to approximate the background on silhouettes)
Sample velocity similar to current pixel velocity (sample and pixel contributing to each other)

IMPROVED SAMPLE CONTRIBUTION
Depth compare function returns two weights (background and foreground) that sum one
Key for proper gradients

float2 DepthCmp(
	float centerDepth,
	float sampleDepth,
	float depthScale )
{
	return saturate( 0.5 + float2( depthScale, -depthScale ) *
		( sampleDepth - centerDepth ) );
}

IMPROVED SAMPLE CONTRIBUTION
Used together as follows to get the weight for a sample:

float SampleWeight(
	float centerDepth, float sampleDepth, float offsetLen, float centerSpreadLen,
	float sampleSpreadLen, float pixelToSampleUnitsScale, float depthScale )
{
	float2 depthCmp = DepthCmp( centerDepth, sampleDepth, depthScale );
	float2 spreadCmp = SpreadCmp( offsetLen, float2( centerSpreadLen, sampleSpreadLen ), pixelToSampleUnitsScale );
	return dot( depthCmp, spreadCmp );
}

PRESORT PASS
We run a pass that we call presort
For each pixel
Calculates its COC
Classifies it as background or foreground
Calculate weights with the alpha for the additive alpha blending average in the main filter pass
Output to a R11G11B10
r: coc
g: SampleAlpha() * DepthCmp2().x
b: SampleAlpha() * DepthCmp2().y
To determine if a pixel is background or foreground we compare with the closest depth in each tile
Presorting before filtering reduces ALU and VGPR pressure on the main filter pass
MAIN PASS
float SpreadToe( float offsetCoc, float spreadCmp )
{
	return	offsetCoc <= 1.0 ?
		pow( spreadCmp, DOF_SPREAD_TOE_POWER ) : spreadCmp;
}
float SpreadCmp(
	float offsetCoc,
	float sampleCoc,
	float pixelToSampleUnitsScale )
{
	return SpreadToe(
		offsetCoc,
		saturate( pixelToSampleUnitsScale * sampleCoc – offsetCoc + 1.0 ) );
}
float spreadCmp = SpreadCmp( offsetCoc, samplePresortParams.coc, spreadScale );
samplePresortParams.backgroundWeight *= spreadCmp;
samplePresortParams.foregroundWeight *= spreadCmp;
background += samplePresortParams.backgroundWeight * float4( sampleColor.rgb, 1.0 );
foreground += samplePresortParams.foregroundWeight * float4( sampleColor.rgb, 1.0 );

OVERVIEW
Being a 60-fps title, the constraints on the shadow filtering are challenging, as there is not budget for many taps
We experimented with randomly per-pixel rotated Poisson disks
We found them to produce low-quality results, with moderate sample counts (eight)
Very unstable under motion

Creating random noise:

values.xy = sv_position;

values.xy = frac( dot( values.xy, magic.xy ) );
…

OVERVIEW
Experimenting and optimizing a noise generator, we found a noise function that we could classify as being half way between dithered and random, and that we called Interleaved Gradient Noise:

float3 magic = float3( 0.06711056, 0.00583715, 52.9829189 );
return frac( magic.z * frac( dot( sv_position, magic.xy ) ) );

Used to rotate the samples:

sincos( 2.0 * PI * InterleavedGradientNoise( sv_position ), rotation.y, rotation.x );
float2x2 rotationMatrix = { rotation.x, rotation.y, -rotation.y, rotation.x };
...
float2 sampleOffset = mul( offsets[i], rotationMatrix );

Best of both worlds:
Produces a rich range of values, similar to random noise
Produces temporally coherent results, similar to dithered schemes
INTERLEAVED GRADIENT NOISE
This noise produces interleaved gradients
This means that objects moving at a constant speed will rotate the samples smoothly

EXPLOITING TEMPORAL SUPERSAMPLING OR DITHERING
If using temporal supersampling (for example SMAA T2x), we can temporally dither the noise:
frameTick.x * InterleavedGradientNoise( sv_position ) + frameTick.y
Where:
frameTick.x = 0.5
frameTick.y = 0.0 or 0.5 for odd and even frames respectively
2x sample count
If the game runs at 60fps, temporal dithering might be enough

UPSAMPLING
Upsampling progressively yields the best results [Kraus2007] [Kawase2011] [Mittring2012] 
Never bilinearly upsample mip6 to mip0 resolution directly
Upsample from mip6 to mip5, from mip5 to mip4 and so on
Why this works? [Kraus2007]
Progressively natural bilinearly upsampling is equivalent to bi-quadratic b-spline filtering
“Natural” means to sample in finer MIP pixel centers
If you box blur at each upscale step (with a , for example), you get a higher order b-spline upscale
UPSAMPLING
Added bonus of upscaling progressively: you can also do the sum with previous mip as you upscale [Mittring2012]
Reduces memory fetches required at higher resolutions

UPSAMPLING - OUR SOLUTION
Progressive upsample (and sum the chain as you upscale) as found in [Mittring2012]
We filter during upsample (on the flight, as in the downsample case):
E’ = E
D’ = D + blur(E’)
C’ = C + blur(D’)
B’ = B + blur(C’)
A’ = A + blur(B’)
We use a 3x3 tent filter



Scaled by a radius parameter
The filter do not map to pixels, has “holes” in it
Repeated convolutions converge to a Gaussian [Kraus2007]
Simple and fast

FIREFLIES
We found that any non-linear intensity mapping before averaging yields produces pulsating artifacts
On small objects that move in steps similar to their size
Linear motion of the objects do not translate to linear color changes
Subpixel motion would solve this issue (multi/supersampling)
But that is not universally available

FIREFLIES - PARTIAL KARIS AVERAGE
We found that doing the Karis average on each  separately mitigates the issue
Intuitively, it allows to recover linear motion of small objects after the fireflies are removed by each separate  filter
